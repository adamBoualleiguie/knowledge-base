# Interview mode: patterns I can defend calmly

If you are using this series to prep for a **Bazel-heavy role**, here is how **I** explain what I did ‚Äî without sounding like I memorized a blog post.

---

## Pattern A ‚Äî ‚ÄúStrangler fig‚Äù migration

> ‚ÄúI kept Docker Compose runtime and Dockerfile registry builds while growing a Bazel graph until CI could block on it.‚Äù

**Evidence:** dual Dockerfile + Bazel OCI matrix; **M4** switched **`bazel_ci`** to **`ci_full.sh`**.

---

## Pattern B ‚Äî ‚ÄúPolicy as data‚Äù

> ‚ÄúOCI bases are digest-pinned in the module; an **allowlist** enforces that every **`oci.pull` name** was reviewed.‚Äù

**Evidence:** `tools/bazel/policy/oci_base_allowlist.txt` + Python checker + **`sh_test`**; first step of **`ci_full.sh`**.

---

## Pattern C ‚Äî ‚ÄúTests routed by tags‚Äù

> ‚ÄúFast PR gates run **`unit`**; integration / e2e / trace are explicit configs.‚Äù

**Evidence:** **`.bazelrc`** **`test:unit`** filters; **`rn_js_checks`**, **`oci_allowlist_test`**, **`frontend:lint`**, etc.

---

## Pattern D ‚Äî ‚ÄúHermeticity is staged‚Äù

> ‚ÄúI used **`sh_test`**, host toolchains, and **`no-sandbox`** where the ROI of perfect hermeticity was low ‚Äî and I kept **Go/proto** paths strict where they shine.‚Äù

**Evidence:** polyglot **`sh_test`** bridges (Elixir, PHP, .NET, RN) vs **`go_binary`** / **`oci_image`** on checkout.

---

## Pattern E ‚Äî ‚ÄúSupply chain instrumentation exists‚Äù

> ‚ÄúRelease workflow emits **SBOM** and runs a **scanner**; failing the build on every CVE is **tunable**.‚Äù

**Evidence:** **`Bazel checkout OCI (release)`** workflow ‚Äî Anchore actions, optional **`oci_push`**.
